The four guarantees · ACID

Transaction

Atomicity

All operations must succeed, or none are applied.

Consistency

Ensures the database remains in a valid state after execution.

Isolation

Guarantees that concurrent transactions do not interfere with each other.

Durability

Once committed, a transaction persists even after a system failure.

Why atomicity matters
04 / 15
Concept 01 · Atomicity

Atomicity

Why atomicity
is important

A transaction bundles several operations into one unit. Either the whole bundle takes effect, or nothing does.

If only one of these operations is executed, the system becomes inconsistent.

Partial Write Problem
05 / 15
Addition 01 · The core risk

Partial Write Problem

One of the most critical issues related to transactions.

Occurs when only a subset of operations in a transaction is executed — due to unexpected failures:

  • System crashes
  • Network errors
  • Application bugs

If a failure occurs in the middle of execution, only some attributes may be updated while others remain unchanged.

This creates an inconsistent state in the database.

Write-Ahead Log
06 / 15
Addition 02 · The safety net

WAL — Write Ahead Log

All changes are first recorded in a log before they are applied to the actual database.

Use the log to either undo incomplete operations or redo completed ones.

Ensures that even if a crash happens mid-execution, the database can be restored to a consistent state.

Who controls the transaction?
07 / 15
Addition 03 · Boundaries

Application vs DB Control

Difference between application-level control and DBMS-level control.

Although a DBMS internally supports transactions, it does not automatically apply them unless explicitly specified.

→ Developers must define transaction boundaries using commands like BEGIN, COMMIT, and ROLLBACK.

Without these commands, each operation may be treated as an independent action — losing the all-or-nothing guarantee.

Case 1 · No safeguards

● Case 1 · Without Transaction

Without Transaction

case1_no_tx.py
# Without Transaction
print("Case 1: WITHOUT Transaction")

try:
    cursor.execute("""
        UPDATE stocks
        SET close = close * 1.1
        WHERE date = (SELECT date FROM stocks LIMIT 1)
    """)

    raise Exception("Simulated Failure!")   # ← crash here

    cursor.execute("""
        UPDATE stocks
        SET volume = volume * 1.1
        WHERE date = (SELECT date FROM stocks LIMIT 1)
    """)
    conn.commit()

except Exception as e:
    print("Error occurred:", e)
    conn.commit()                          # commits anyway!
  • Update stock price (close value).
  • An intentional error is raised before updating trading volume.
  • The first update was applied successfully, but the second update was never executed.

Case 2 · With safeguards
11 / 15
● Case 2 · With Transaction

With Transaction

case2_with_tx.py
# With Transaction
print("Case 2: WITH Transaction")

try:
    conn.execute("BEGIN")                     # ← open transaction

    cursor.execute("""
        UPDATE stocks
        SET close = close * 1.1
        WHERE date = (SELECT date FROM stocks LIMIT 1)
    """)

    raise Exception("Simulated Failure!")

    cursor.execute("""
        UPDATE stocks
        SET volume = volume * 1.1
        WHERE date = (SELECT date FROM stocks LIMIT 1)
    """)
    conn.commit()

except Exception as e:
    print("Error occurred:", e)
    conn.rollback()                        # ← undo everything
  • Applied transaction control around the same operations.
  • Triggered the exact same error again.
  • When the error occurred, the system rolled back the entire transaction.
